## Task Breakdown

### Task 0: Sanity check da demo live (30 min)
1. Executar `./run-all.sh` do zero — confirmar saída limpa (professor pode pedir execução live)
2. Executar `pytest tests/` (via .venv) — 14/14 passing
3. **Teste de clone fresco REAL** (não apenas ler o README): `git clone` do repo GitHub em diretório temporário e seguir o README literalmente (criar venv → instalar deps → `./run-all.sh`). Este é o único caminho que valida o risco do `.venv/bin/python` hardcoded. Se falhar: corrigir README e/ou run-all.sh ANTES da defesa.
4. Confirmar CSVs em `results/` e PNGs em `benchmark-plots/` commitados (contingência se live falhar)

**Success:** benchmark + testes rodam limpos; clone fresco reproduz o fluxo do README; contingência disponível no repo.

### Task 1: Roteiro de apresentação (1–1.5 h)
Criar `.planning/phases/04-defense/ROTEIRO.md`:
- Speaker notes por slide (10 slides), ~1.5 min cada, frases curtas e diretas
- Abertura (2 min): quem somos, o que fizemos, qual a contribuição
- Por slide: mensagem central (1 frase), pontos de apoio (2–4), transição para o próximo
- Slide 7–9 (resultados): narrativa "o que o gráfico mostra → por quê → o que concluímos"
- Fechamento: contribuição inédita explícita (framework entropia + repetição lexical) + convite a perguntas
- Marcações de timing acumulado (checkpoint aos 5, 10, 13 min)

**Success:** roteiro completo, timing soma ≤15 min, contribuição inédita nomeada.

### Task 2: Banco de Q&A antecipado (1.5–2 h)
Criar `.planning/phases/04-defense/QA-BANK.md` com ≥20 perguntas prováveis + respostas fundamentadas (números EXATOS do artigo). Categorias:
1. **Algoritmos (6+):** como Huffman constrói a árvore? por quê min-heap? como LZW reconstrói o dicionário sem transmiti-lo? caso especial do LZW na decodificação? por quê limite 4096 entradas? por quê códigos de 16 bits?
2. **Complexidade (4+):** O(n log n) de onde vem? por quê O(m) amortizado no LZW? complexidade de espaço de cada um? teoria bateu com a prática?
3. **Resultados (6+):** por quê Huffman vence em alta entropia? por quê LZW expande arquivos pequenos? por quê LZW domina no log (6.76)? por quê C favorece LZW? qual algoritmo escolher para código-fonte? trade-off memória (110 vs 871 KB)?
4. **Metodologia (4+):** por quê SHA-256? por quê entropia de Shannon + repetição lexical? por quê esses 6 arquivos? limitações do estudo?
5. **Níveis de aprendizagem (VAL-07):** resposta modelo conectando ANÁLISE (comparou), AVALIAÇÃO (concluiu qual melhor por perfil), CRIAÇÃO (implementou + framework de benchmark)
6. **Seção final obrigatória — "Pergunta sem resposta preparada":** protocolo de resposta honesta (reconhecer limite do estudo, conectar ao que FOI medido, propor como trabalho futuro). Ex.: "Não medimos X; nosso escopo foi Y; seria extensão natural."

**Success:** ≥20 Q&A, números conferidos contra o artigo, VAL-07 mapeado, seção de fallback presente.

### Task 3: Plano de contingência da demo (30 min)
Criar `.planning/phases/04-defense/DEMO-CHECKLIST.md`:
- Pré-defesa: repo público confirmado, `run-all.sh` testado na véspera, browser com repo aberto em aba
- Roteiro da demo live (2–3 min): mostrar estrutura src/, rodar run-all.sh OU abrir results/ + benchmark-plots/ prontos
- Contingência nível 1: execução falha → mostrar CSVs/PNGs commitados
- Contingência nível 2: sem internet → clone local + PDFs offline
- Itens de backup: PDFs do artigo e slides em 2 dispositivos + pendrive

**Success:** demo tem caminho feliz + 2 níveis de fallback documentados.

### Task 4: Checklist dia-da-defesa (30 min)
Criar `.planning/phases/04-defense/DIA-DA-DEFESA.md`:
- **VAL-01 (crítico):** confirmação de presença de TODOS os integrantes (nota zero se faltar) — combinar horário/local com folga
- Logística: chegar 15 min antes, testar projetor/áudio, slides abertos
- Postura: falar para a banca, não para o slide; divisão de falas entre integrantes definida
- Pós-slides: quem responde qual categoria de pergunta (dividir Q&A bank por integrante)
- Ensaio: ≥2 ensaios cronometrados completos antes da data (registrar tempos)

**Success:** checklist acionável; divisão de responsabilidades entre integrantes explícita.
