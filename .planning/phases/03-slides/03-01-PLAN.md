## Task Breakdown

### Task 0: Prerequisite Check (5 min)
1. `kpsewhich beamer.cls` → já verificado ✓ (presente no basictex 2026)
2. `kpsewhich beamerthememetropolis.sty` → se ausente, usar tema `Madrid` (built-in)
3. Criar diretório `slides/` e COPIAR as figuras do artigo para dentro dele: `cp article/fig-ratio.png article/fig-tempo.png article/fig-memoria.png slides/`
   — Fonte canônica é `article/fig-*.png` (as figuras exatas usadas no artigo). NÃO usar path relativo `../benchmark-plots/`: benchmark re-runs podem divergir silenciosamente do artigo (git status já mostra `tempo-execucao.png` modificado).

**Success:** Beamer disponível, 3 figuras presentes, `slides/` criado.

### Task 1: Criar slides.tex — 10 frames (2–3 h)
Escrever `slides/slides.tex` (Beamer, 12pt ou padrão, `[utf8]{inputenc}` + `[T1]{fontenc}` + `[brazilian]{babel}`).

**Conteúdo por frame (fonte: artigo.tex Phase 2 — números EXATOS do artigo):**

1. **Capa/Grupo:** título do trabalho, `PLACEHOLDER: nomes e matrículas`, disciplina AED-II (UFABC), link GitHub
2. **Problema (1/2):** por quê comprimir código-fonte — crescimento de repositórios, backups/CI/monorepos, custo de armazenamento e transferência
3. **Problema (2/2):** dualidade do código-fonte (distribuição desigual de bytes + tokens repetidos); 4 questões de pesquisa resumidas
4. **Trabalhos relacionados:** Huffman (1952), Ziv-Lempel (1977, 1978), Welch (1984), Shannon (1948), comparações recentes (Nambiar 2025, Shrividhiya 2021) — 1 linha cada
5. **Algoritmo Huffman:** min-heap + árvore binária, códigos por frequência, O(n log n) + O(m), forte em distribuição desigual, fraco em padrões longos
6. **Algoritmo LZW:** dicionário adaptativo (hash, 4096 entradas), códigos 16 bits, O(m) amortizado, dicionário reconstruído na decodificação, forte em repetição
7. **Experimentos — taxa de compressão:** figura `compression-ratio.png` + destaque: Huffman vence Python médio/alta entropia; LZW vence C e log (6.76!)
8. **Experimentos — tempo:** figura `tempo-execucao.png` + destaque: compressão similar; descompressão LZW 2.5× mais rápida em arquivos médios
9. **Experimentos — memória:** figura `memoria.png` + destaque: Huffman 5–10× mais eficiente (110 KB vs 871 KB); LZW expande arquivos pequenos/alta entropia
10. **Conclusões:** sem vencedor absoluto — Huffman: memória/entropia/pequenos; LZW: repetição/descompressão rápida; futuras pesquisas (repositórios completos, Git, variantes híbridas); link GitHub repetido

**Regras de conteúdo:**
- Máx ~6 bullets/frame, sem parágrafos
- Números copiados EXATOS do artigo (não recalcular)
- Figuras: `\includegraphics[height=0.62\textheight]{fig-ratio.png}` etc. — cópias locais em `slides/` (Task 0)
- PT-BR norma culta com acentuação correta NO TEXTO; nomes de arquivos/labels SEMPRE ASCII (lição Phase 2)
- Sem citações novas — só as do artigo

**Success:** slides.tex com exatamente 10 frames, conteúdo mapeado 1:1 com exigência do professor.

### Task 2: Compilar + iterar layout (1–2 h)
1. `cd slides && pdflatex -interaction=nonstopmode slides.tex` (2 passes)
2. Verificar: exatamente 10 páginas no PDF (`Output written on slides.pdf (10 pages, ...)`)
3. Inspecionar cada frame via `pdftotext` + verificação de overflow no log (`Overfull \vbox` E `Overfull \hbox` — URLs longas como o link GitHub tipicamente estouram hbox)
4. Ajustar: reduzir bullets, `\small` em tabelas se necessário, reposicionar figuras
5. Renomear output final: `apresentacao.pdf`

**Success:** PDF com 10 páginas, zero overflow warnings relevantes, figuras renderizadas.

### Task 3: Audit visual + conteúdo (30–60 min)
1. Extrair texto do PDF e conferir: acentuação correta, números idênticos aos do artigo
2. Conferir presença: nomes/disciplina (slide 1), link GitHub (slides 1 e 10), 3 figuras (slides 7–9)
3. Conferir legibilidade: sem bullets cortados, sem texto sobreposto (checar log por overfull)
4. Timing sanity check: 10 slides × 1.5 min ≈ 15 min ✓

**Success:** Todos os 9 requisitos SLD verificáveis no PDF final.

### Task 4: Cross-check final + prontidão Moodle (30 min)
1. Tabela SLD-01–09: ✓/⚠/✗ com evidência
2. Registrar pendência humana: substituir placeholder de nomes/matrículas
3. Atualizar `03-AUDIT.md`, criar registro de conclusão
4. Commit `.planning/` (verificar se `slides/` é versionável — `article/` está no .gitignore; decidir: adicionar `slides/` ao repo se git aceitar, senão documentar)

**Success:** SLD-01–09 mapeados, pendências registradas, estado commitado.

**GATE (hard):** Fase NÃO é "submission-ready" enquanto o placeholder de nomes/matrículas existir. Status final da fase = "complete pending human input" até nomes reais preenchidos e PDF recompilado.
