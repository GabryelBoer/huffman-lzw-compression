Add strategic documentation to LZW modules following research guidelines:

**src/lzw/encoder.py:**
- Module docstring: explain LZW algorithm (dictionary-based substring replacement, streaming, O(n) time)
- Highlight: dictionary is hash table capped at 4096 entries; reset on overflow
- Each function:
  - Docstring: purpose, Args, Returns, Complexity (Time O(n), Space O(d) where d=dict size)
  - Example: "encode(data) -> BitStream: Replace substrings with dictionary codes. O(n) single pass. Space O(d), d ≤ 4096."
- Inline comments at decision points:
  - Dictionary insertion: explain why hash (O(1) lookup) vs tree (would be O(log d))
  - Code output: explain variable-length int encoding if custom (vs fixed 12-bit codes)
  - Dictionary overflow: comment capping logic (reset to 256 base symbols)
  - Longest match: explain greedy approach (current best compression)
- Avoid: obvious loop explanations

**src/lzw/decoder.py:**
- Module docstring: explain LZW decoding (reverse engineer dictionary from codes)
- Challenge: decoder must rebuild dictionary from codes alone (no explicit dictionary transmitted)
- Functions: docstrings with purpose, complexity (O(n) time, O(d) space)
- Inline: comment special case handling (e.g., code > current dictionary size edge case)
- Explain: why decoder mirrors encoder dictionary construction
