**Section 3 — Algorithms at a Glance (2 subsections):**
- **Huffman:** Frequency table → Heap → Binary tree → Variable-length codes
  - Time: O(n log n) build + O(n) encode/decode
  - Space: O(alphabet size, typically 256 for bytes)
  - Note: Optimal for high-entropy data (random byte distributions)
  
- **LZW:** Dictionary-based substring replacement
  - Time: O(n) single pass (hash dictionary)
  - Space: O(dictionary size, capped at 4096 entries in implementation)
  - Note: Strong on repetitive patterns (code with duplicate expressions)

**Table 1 — Theoretical Complexity:**
| Algorithm | Phase | Operation | Time Complexity | Space Complexity | Rationale |
| --- | --- | --- | --- | --- | --- |
| Huffman | Build | Frequency count + heap insertion | O(n + k log k) | O(k) | n=input, k=unique symbols; heap ops O(log k) |
| Huffman | Build | Tree traversal for codes | O(k) | O(k) | Linear walk of k leaves |
| Huffman | Encode | Symbol lookup + bitwrite | O(n) | O(1) | Each symbol O(1) dict lookup + O(1) amortized write |
| Huffman | Decode | Bit read + tree hop | O(n) | O(1) | Each output traverses tree (depth ≤ log k) |
| LZW | Encode | Dict insert + match | O(n) | O(d) | d=dict size (4096); hash O(1) amortized |
| LZW | Decode | Dict insert + code lookup | O(n) | O(d) | Same as encode |
